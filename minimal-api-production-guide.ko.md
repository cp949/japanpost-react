# 운영용 Japan Post 연동 서버 개발 가이드

이 문서는 다른 프로젝트에서 `apps/minimal-api`에 준하는 운영 서버를 구현할 때
참고하는 한글 가이드다. 목적은 로컬 참고 구현을 그대로 복제하는 것이 아니라,
운영 환경에서 Japan Post 연동 책임을 담당하는 서버를 안정적으로 재구현하는 데
필요한 요구사항과 판단 기준을 정리하는 것이다.

이 저장소의 `apps/minimal-api`는 로컬 검증용 참고 구현이다. 운영 서버는 이 코드의
핵심 책임과 외부 계약은 유지하되, 보안 정책, 관측성, 배포 방식, 장애 대응은 각
프로젝트의 운영 기준에 맞게 강화해야 한다.

## 1. 이 서버가 맡아야 하는 역할

운영 서버는 브라우저와 Japan Post API 사이의 얇은 프록시가 아니라, 다음 책임을
가지는 서버 사이드 어댑터다.

- Japan Post 인증 정보 보관과 토큰 발급
- 업스트림 API 호출과 타임아웃 제어
- 클라이언트 입력 검증
- 업스트림 응답을 프런트엔드 친화적인 형식으로 정규화
- 업스트림 오류를 서비스 정책에 맞는 HTTP 오류로 변환
- 호출량 제어, 로깅, 모니터링, 보안 정책 적용

브라우저가 Japan Post API를 직접 호출하게 두면 비밀값 노출, CORS, 토큰 관리,
에러 처리 일관성 문제가 생기므로, 운영 환경에서는 반드시 서버에서 중계한다.

## 2. 권장 아키텍처

기본 흐름은 아래와 같다.

```text
브라우저 / 웹앱
  -> 운영 API 서버
  -> Japan Post 인증 토큰 발급
  -> Japan Post 주소 조회 API 호출
  -> 운영 API 서버가 응답 정규화 후 반환
```

권장 분리는 다음과 같다.

- HTTP 라우터 계층
  - 요청 파싱, 인증/권한, CORS, 응답 헤더, HTTP 상태 코드 결정
- Japan Post 어댑터 계층
  - 토큰 취득, 토큰 캐시, 업스트림 호출, 업스트림 오류 해석
- 응답 정규화 계층
  - Japan Post 원본 필드를 프런트엔드 계약에 맞는 주소 객체로 변환
- 운영 공통 계층
  - 로깅, 추적 ID, 메트릭, 레이트 리밋, 설정 주입

`apps/minimal-api/src/server.ts`와
`apps/minimal-api/src/japanPostAdapter.ts`의 분리는 그대로 참고할 만하다. 운영
프로젝트에서도 HTTP 처리와 업스트림 연동 로직을 분리하는 편이 유지보수에 유리하다.

## 3. 외부 API 계약

프런트엔드와의 계약을 안정적으로 유지하려면, 운영 서버에서도 외부 인터페이스를
명시적으로 고정하는 편이 좋다. 현재 참고 구현이 제공하는 계약은 다음과 같다.

### 3.1 `GET /health`

서버가 실제 조회 요청을 받을 준비가 되었는지 확인한다. 단순 프로세스 생존 여부가
아니라, Japan Post 인증 자격정보가 유효하고 토큰 발급까지 가능한지를 확인하는
readiness 성격의 엔드포인트로 두는 것을 권장한다.

정상 응답 예시:

```json
{ "ok": true }
```

비정상 응답 예시:

```json
{ "ok": false, "error": "..." }
```

운영 환경에서는 이 엔드포인트를 다음 용도로 활용할 수 있다.

- 컨테이너 readiness probe
- 배포 직후 smoke test
- 장애 조치 전환 기준

### 3.2 `GET /searchcode/:code`

7자리 우편번호로 주소를 조회한다. 입력은 하이픈 제거 후 검증하는 편이 안전하다.
참고 구현은 숫자만 남긴 뒤 7자리가 아니면 `400`을 반환한다.

응답 예시:

```json
{
  "postalCode": "1020072",
  "addresses": [
    {
      "postalCode": "1020072",
      "prefecture": "東京都",
      "prefectureKana": "トウキョウト",
      "city": "千代田区",
      "cityKana": "チヨダク",
      "town": "飯田橋",
      "townKana": "イイダバシ",
      "address": "東京都 千代田区 飯田橋",
      "provider": "japan-post"
    }
  ]
}
```

### 3.3 `GET /addresszip?q=<query>`

자유어 검색을 수행한다. 참고 구현은 내부적으로 Japan Post `addresszip` API를 `POST`
로 호출하지만, 프런트엔드 계약은 단순한 `GET`으로 노출한다. 운영 환경에서도
클라이언트 계약은 단순하게 유지하고, 업스트림 프로토콜 차이는 서버 내부에서
흡수하는 구성이 좋다.

응답 예시:

```json
{
  "query": "千代田",
  "addresses": [
    {
      "postalCode": "1020072",
      "prefecture": "東京都",
      "prefectureKana": "トウキョウト",
      "city": "千代田区",
      "cityKana": "チヨダク",
      "town": "飯田橋",
      "townKana": "イイダバシ",
      "address": "東京都 千代田区 飯田橋",
      "provider": "japan-post"
    }
  ]
}
```

## 4. 응답 정규화 규칙

Japan Post 원본 응답을 그대로 노출하지 말고, 프런트엔드에서 바로 사용할 수 있는
정규화된 주소 객체로 변환하는 것을 권장한다. 참고 구현은 아래 필드를 만든다.

- `postalCode`
- `prefecture`
- `prefectureKana`
- `city`
- `cityKana`
- `town`
- `townKana`
- `address`
- `provider`

핵심 규칙은 다음과 같다.

- 우편번호는 숫자 7자리 문자열로 통일한다.
- 공백과 `null`을 정리해 빈 문자열 또는 `undefined`로 정규화한다.
- `address`는 `prefecture + city + town + 상세주소`를 조합한 사람이 읽기 좋은
  문자열로 만든다.
- 업스트림 원본 필드명이 바뀌더라도, 프런트엔드 계약은 가능한 한 유지한다.

운영 프로젝트에서 프런트엔드 계약이 이미 정해져 있다면, 이 정규화 계층에서만
맵핑을 바꾸고 나머지 계층은 그대로 두는 구조가 좋다.

## 5. 인증과 토큰 캐시 전략

`apps/minimal-api/src/japanPostAdapter.ts`의 가장 중요한 책임은 인증과 토큰 재사용
이다. 운영 서버에서도 같은 전략이 필요하다.

### 5.1 필수 설정

- `JAPAN_POST_CLIENT_ID`
- `JAPAN_POST_SECRET_KEY`

### 5.2 선택 설정

- `JAPAN_POST_BASE_URL`
- `JAPAN_POST_TOKEN_PATH`
- `JAPAN_POST_SEARCH_CODE_PATH`
- `JAPAN_POST_ADDRESS_ZIP_PATH`
- `JAPAN_POST_X_FORWARDED_FOR`

### 5.3 구현 원칙

- 토큰은 메모리 캐시 또는 공용 캐시에 저장한다.
- 만료 직전까지 쓰지 말고, 여유 시간을 두고 미리 갱신한다.
- 동시에 여러 요청이 들어와도 토큰 재발급이 한 번만 일어나도록 in-flight 요청을
  합친다.
- 업스트림이 `401`을 반환하면 캐시를 비우고 한 번만 재시도한다.
- 토큰 발급과 API 호출에는 각각 타임아웃을 둔다.

단일 인스턴스 서버라면 프로세스 메모리 캐시로도 충분할 수 있다. 다중 인스턴스
환경에서 토큰 발급 비용이나 레이트 리밋이 민감하다면 Redis 같은 공용 캐시를 검토할
수 있다. 다만 구현 복잡도가 늘어나므로, 먼저 메모리 캐시로 시작하고 필요 시
분리하는 접근이 현실적이다.

## 6. 입력 검증과 오류 매핑

운영 서버는 업스트림 오류를 그대로 중계하지 말고, 서비스 관점에서 의미 있는
오류로 정리해야 한다.

권장 기준은 다음과 같다.

- 잘못된 우편번호 형식: `400`
- 빈 검색어: `400`
- 주소 없음: `404`
- 업스트림 인증 실패: `502` 또는 운영 정책에 맞는 서버 오류
- 업스트림 타임아웃: `504`
- 예상하지 못한 서버 오류: `500`

중요한 점은 "클라이언트가 고칠 수 있는 오류"와 "운영 서버 또는 업스트림 문제"를
분리하는 것이다. 프런트엔드는 이 구분을 기준으로 재시도, 사용자 메시지, 에러 로그
정책을 다르게 가져갈 수 있다.

오류 응답은 최소한 아래 형식을 유지하는 편이 좋다.

```json
{ "error": "사용자 또는 운영 로그에 남길 수 있는 메시지" }
```

운영 환경에서는 여기에 내부 추적용 `requestId`를 추가하는 것도 권장한다.

## 7. CORS, 인증, 보안 주의사항

참고 구현은 로컬 데모 편의를 위해 `access-control-allow-origin: "*"`를 사용한다.
이 설정은 운영 서버에 그대로 쓰면 안 된다.

운영 환경에서는 다음 원칙을 적용한다.

- 허용 origin을 명시적으로 제한한다.
- 필요한 메서드와 헤더만 허용한다.
- 브라우저 인증 전략이 있다면 쿠키/세션/CORS 정책을 함께 설계한다.
- 시크릿은 `.env` 파일 커밋이 아니라 비밀 저장소나 배포 플랫폼 시크릿 기능으로
  관리한다.
- 업스트림 응답 전문이나 민감한 자격정보는 로그에 남기지 않는다.
- 레이트 리밋과 abuse 방어 정책을 넣는다.

사내 서비스이든 공개 서비스이든, 이 서버는 외부 주소 조회를 대리 수행하는 진입점이
되므로 최소한의 남용 방지 장치는 필요하다.

## 8. 운영 관측성과 장애 대응

운영 서버에선 "조회가 안 된다"는 문제를 빠르게 분리 진단할 수 있어야 한다.
최소한 아래 항목은 준비하는 것을 권장한다.

- 요청 수, 성공률, 오류율 메트릭
- 업스트림 토큰 발급 실패 횟수
- 업스트림 주소 조회 실패 횟수
- 업스트림 응답 시간과 타임아웃 횟수
- 애플리케이션 로그의 요청 ID
- 배포 직후 `/health` 확인 절차

로그에는 아래 정도가 남으면 유용하다.

- 내부 요청 ID
- 호출 엔드포인트
- 응답 상태 코드
- 처리 시간
- 업스트림 요청 실패 여부

단, 검색어 전체나 원본 주소 응답을 무분별하게 남기면 개인정보 취급 이슈가 생길 수
있으므로, 로그 마스킹 기준을 먼저 정하는 편이 좋다.

## 9. 운영 구현 시 권장 체크리스트

구현 시작 전:

- 외부 공개 API 경로를 확정했다.
- 프런트엔드가 기대하는 응답 스키마를 확정했다.
- 시크릿 저장 위치를 정했다.
- 허용 origin 목록을 정했다.

구현 중:

- 토큰 캐시와 만료 버퍼를 구현했다.
- `401` 재인증 재시도를 한 번만 수행한다.
- 입력 검증과 오류 매핑이 엔드포인트마다 일관적이다.
- 업스트림 타임아웃을 설정했다.
- 구조화 로그와 요청 ID를 넣었다.

배포 전:

- `/health`가 실제 토큰 발급 가능 여부를 반영한다.
- 잘못된 입력에 `400`, 결과 없음에 `404`를 반환한다.
- 시크릿이 로그나 응답에 노출되지 않는다.
- wildcard CORS가 제거되었다.
- 레이트 리밋 또는 동등한 보호 장치가 적용되었다.
- 운영 모니터링과 알람 조건을 연결했다.

## 10. 참고 구현에서 가져와도 되는 것과 바꿔야 하는 것

그대로 참고해도 되는 것:

- HTTP 계층과 어댑터 계층 분리
- 토큰 캐시와 만료 버퍼
- `401` 발생 시 토큰 무효화 후 1회 재시도
- 우편번호/검색어 입력 검증
- 프런트엔드 친화적인 응답 정규화

운영에 맞게 반드시 바꿔야 하는 것:

- `access-control-allow-origin: "*"`
- 로컬 전용 포트/실행 스크립트 전제
- 최소 수준의 로그 출력만 있는 상태
- 단순 프로세스 메모리만 쓰는 캐시 전략
- 데모 편의 위주의 에러 메시지 표현

## 11. 권장 구현 순서

다른 프로젝트에서 새로 만들 때는 아래 순서가 가장 안전하다.

1. 외부 API 계약과 응답 스키마를 먼저 고정한다.
2. Japan Post 어댑터를 별도 모듈로 구현한다.
3. 토큰 발급과 캐시를 완성한다.
4. `/health`, `/searchcode/:code`, `/addresszip` 순서로 라우트를 붙인다.
5. 입력 검증과 오류 매핑을 정리한다.
6. 로그, 메트릭, CORS, 레이트 리밋을 붙인다.
7. 운영 환경에서 smoke test와 readiness probe를 연결한다.

## 12. 이 저장소에서 확인할 파일

운영 서버 설계 시 아래 파일이 직접적인 참고 자료다.

- `apps/minimal-api/src/server.ts`
- `apps/minimal-api/src/japanPostAdapter.ts`
- `japanpost-development-guide.md`

이 세 파일을 함께 보면, 현재 저장소가 최소한 어떤 외부 계약과 에러 처리 규칙을
유지하려 하는지 빠르게 파악할 수 있다.
